### Source of Truth

-**สตริง UI**: `src/i18n/messages/en.json` (แหล่งที่มาภาษาอังกฤษ ~2800 คีย์) -**ไฟล์โลคัล**: `src/i18n/messages/{locale}.json` (แปล 30 รายการ) -**Framework**: `next-intl` พร้อมการแก้ไขโลแคลตามคุกกี้ -**Config**: `src/i18n/config.ts` — กำหนดสถานที่ ชื่อภาษา ธงทั้งหมด 30 รายการ### Runtime Flow

1. ผู้ใช้เลือกภาษา → ชุดคุกกี้ `NEXT_LOCALE`
2. `src/i18n/request.ts` แก้ไขภาษา: คุกกี้ → ส่วนหัว `ยอมรับภาษา` → ทางเลือกสำรอง `en`
3. โหลดการนำเข้าแบบไดนามิก `messages/{locale}.json`
4. ส่วนประกอบใช้ `useTranslations("namespace")` และ `t("key")`### Supported Locales

### 1. Register the Locale

แก้ไข `src/i18n/config.ts`:```ts
// Add to LOCALES array
"xx",
// Add to LANGUAGES array
{ code: "xx", label: "XX", name: "Language Name", flag: "🏳️" },

````

### 2. Add to Generator

แก้ไข `scripts/i18n/generate-multilang.mjs` — เพิ่มรายการใน `LOCALE_SPECS`:```js
{
  code: "xx",
  googleTl: "xx",
  label: "XX",
  flag: "🏳️",
  languageName: "Language Name",
  readmeName: "Language Name",
  docsName: "Language Name",
},
````

### 3. Generate Initial Translation

```bash
node scripts/i18n/generate-multilang.mjs messages
```

สิ่งนี้จะสร้าง `src/i18n/messages/xx.json` แปลอัตโนมัติจาก `en.json` ผ่าน Google Translate### 4. Review & Fix Auto-Translations

การแปลอัตโนมัติเป็นจุดเริ่มต้น ตรวจสอบด้วยตนเองสำหรับ:

- ความแม่นยำทางเทคนิค
- คำศัพท์ที่เหมาะสมกับบริบท
- การจัดการตัวยึดตำแหน่งอย่างเหมาะสม (`{count}`, `{value}` ฯลฯ)### 5. Validate

```bash
python3 scripts/validate_translation.py quick -l xx
python3 scripts/validate_translation.py diff common -l xx
```

### 6. Generate Translated Documentation

```bash
node scripts/i18n/generate-multilang.mjs docs
```

## Auto-Translation Pipeline

### generate-multilang.mjs (Google Translate)

**เครื่องมือแปลอัตโนมัติหลัก**— ใช้ API ฟรีของ Google Translate เพื่อสร้างการแปลสำหรับสตริง UI, README และเอกสารประกอบ```bash
node scripts/i18n/generate-multilang.mjs [messages|readme|docs|all]

````

| โหมด | มันทำอะไร |
| ---------- | ----------------------------------------------------------------------------- |
| `ข้อความ` | แปลคีย์ที่หายไปใน `src/i18n/messages/{locale}.json` จาก `en.json` |
| `อ่านหน่อยสิ'' | แปล `README.md` เป็นภาษาทั้งหมดเป็น `README.{code}.md` ในโปรเจ็กต์ root |
| `เอกสาร` | แปล `DOC_SOURCE_FILES` เป็น `docs/i18n/{locale}/{docName}` |
| `ทั้งหมด` | รันทั้งสามโหมด |

**คุณสมบัติ:**

-**การป้องกันข้อความ**: มาสก์โค้ดบล็อก (` ``` `), โค้ดอินไลน์ (`` ` ``), ลิงก์/รูปภาพมาร์กดาวน์ (`[ข้อความ](url)`), แท็ก HTML, ตาราง และตัวยึดตำแหน่ง ICU (`{count}`, `{value}`, `{total}` ฯลฯ) ก่อนการแปล จากนั้นจึงกู้คืน
-**การรวมกลุ่มเป็นกลุ่ม**: รวมหลายสตริงด้วยตัวคั่น `__OMNIROUTE_I18N_SEPARATOR__` เพื่อลดการเรียก API ให้เหลือน้อยที่สุด (สูงสุด 1,800 ตัวอักษรต่อคำขอ)
-**แคชในหน่วยความจำ**: หลีกเลี่ยงการเรียก API ซ้ำซ้อนสำหรับสตริงที่ซ้ำกันภายในเซสชัน
-**ตรรกะลองอีกครั้ง**: การย้อนกลับแบบเอ็กซ์โปเนนเชียล (สูงสุด 5 ครั้งโดยมีความล่าช้า 300ms × พยายามดีเลย์) สำหรับข้อผิดพลาด 429/5xx
-**หมดเวลา**: 20 วินาทีต่อคำขอ
-**ข้ามที่มีอยู่**: หากมีไฟล์เป้าหมายอยู่แล้ว ไฟล์นั้นจะไม่ถูกเขียนทับ

**พฤติกรรมที่สำคัญ:**

- `docs/i18n/README.md` ได้รับการ**สร้างใหม่**แต่ละครั้ง — เป็นดัชนีที่สร้างขึ้นอัตโนมัติสำหรับเอกสารทั้งหมด
- ไฟล์รูท `README.{code}.md` จะถูกสร้างขึ้นเฉพาะในกรณีที่ไม่มีอยู่ (ข้ามภาษาใน `EXISTING_README_CODES`)
- แถบภาษา (`🌐**ภาษา:**...`) จะถูกแทรก/อัปเดตโดยอัตโนมัติในเอกสารที่แปลทั้งหมด### i18n_autotranslate.py (LLM-based)

**นักแปลรอง**— ใช้ LLM API ที่เข้ากันได้กับ OpenAI (รวมถึง OmniRoute เองด้วย) เพื่อแปลไฟล์มาร์กดาวน์ `docs/i18n/` ที่มีอยู่ ดีที่สุดสำหรับการขัดเกลาหรือแปลเอกสารใหม่ที่มีคุณภาพดีกว่า Google แปลภาษา```bash
python3 scripts/i18n_autotranslate.py \
  --api-url http://localhost:20128/v1 \
  --api-key sk-your-key \
  --model gpt-4o
````

**คุณสมบัติ:**

- สแกนไฟล์มาร์กดาวน์ `docs/i18n/` สำหรับย่อหน้าภาษาอังกฤษ
- ข้ามบล็อกโค้ด ตาราง และเนื้อหาที่แปลแล้ว
- ส่งย่อหน้าไปยัง LLM พร้อมแจ้งระบบการแปลทางเทคนิค
- รองรับทั้งหมด 30 ภาษา## Validation & QA

### validate_translation.py

**เครื่องมือตรวจสอบการแปล**— เปรียบเทียบ JSON ของสถานที่ใดๆ กับ `en.json` และรายงานปัญหา```bash

# Quick check (counts only)

python3 scripts/validate_translation.py quick -l cs

# Output:

# Missing: 0

# Untranslated: 0

# Ignored (UNTRANSLATABLE_KEYS): 236

# Detailed diff by category

python3 scripts/validate_translation.py diff common -l cs
python3 scripts/validate_translation.py diff settings -l cs

# Export to CSV

python3 scripts/validate_translation.py csv -l cs > report.csv

# Export to Markdown

python3 scripts/validate_translation.py md -l cs > report.md

# Full report (default)

python3 scripts/validate_translation.py -l cs

`````

**การตรวจจับ:**

-**คีย์หายไป**— คีย์ใน `en.json` แต่ไม่อยู่ในไฟล์โลแคล
-**คีย์พิเศษ**— คีย์ในไฟล์โลแคลแต่ไม่ใช่ใน `en.json`
-**คีย์ที่ไม่ได้แปล**— คีย์ที่มีค่าภาษาเท่ากับต้นฉบับภาษาอังกฤษ (ไม่รวมรายการที่อนุญาต)
-**ตัวยึดตำแหน่งไม่ตรงกัน**— ตัวยึดตำแหน่ง ICU ที่ไม่ตรงกันระหว่างแหล่งที่มาและการแปล

**รหัสออก:**
| รหัส | ความหมาย |
|------|---------|
| 0 | ตกลง |
| 1 | ข้อผิดพลาดทั่วไป |
| 2 | สตริงหายไป (ข้อผิดพลาดอย่างหนัก) |
| 3 | คำเตือนที่ไม่ได้แปล (อ่อน) |

**สภาพแวดล้อม:**ตั้งค่า `TRANSLATION_LANG=cs` หรือใช้แฟล็ก `-l cs`### check_translations.py

**ตัวตรวจสอบคีย์ Code-to-JSON**— สแกน `src/**/*.tsx` และ `src/**/*.ts` สำหรับการเรียก `useTranslations()` และยืนยันว่าคีย์ที่อ้างอิงทั้งหมดมีอยู่ใน `en.json````bash
# Basic check
python3 scripts/check_translations.py

# Verbose output
python3 scripts/check_translations.py --verbose

# Auto-fix (adds missing keys to en.json)
python3 scripts/check_translations.py --fix
`````

### generate-qa-checklist.mjs

**การวิเคราะห์แบบคงที่ QA**— สแกนไฟล์เพจ Next.js เพื่อหาตัวชี้วัดความเสี่ยง i18n และสร้างรายงาน Markdown```bash
node scripts/i18n/generate-qa-checklist.mjs

````

**การตรวจสอบ:**

- การใช้งานคลาสที่มีความกว้างคงที่ (ความเสี่ยงล้น)
- คลาสทิศทางซ้าย/ขวา (ความเสี่ยง RTL)
- รูปแบบที่มีแนวโน้มว่าจะถูกตัด
- ความเท่าเทียมกันของสถานที่ (คีย์หายไป/คีย์พิเศษเทียบกับ `en.json`)
- แถบตัวเลือกภาษา README ในตำแหน่งที่มีลำดับความสำคัญ (`es`, `fr`, `de`, `ja`, `ar`)

**ผลลัพธ์:**`docs/reports/i18n-qa-checklist-{date}.md`### run-visual-qa.mjs

**Visual QA ผ่าน Playwright**— จับภาพหน้าจอของเส้นทางแดชบอร์ดทั้งหมดในหลายสถานที่และวิวพอร์ต จากนั้นประเมินความสมบูรณ์ของหน้า```bash

````

**การตรวจจับ:**

- ข้อความล้น
- การตัดองค์ประกอบ
- เค้าโครง RTL ไม่ตรงกัน

**เอาต์พุต:**`docs/reports/i18n-visual-qa-{date}.md` + รายงาน JSON## Managing Untranslatable Keys

### untranslatable-keys.json

**ไฟล์:**`scripts/i18n/untranslatable-keys.json`

รายการที่อนุญาตของคีย์ที่ควรเหมือนกับต้นฉบับภาษาอังกฤษ ใช้โดย `validate_translation.py` เพื่อหลีกเลี่ยงคำเตือน "ไม่ได้แปล" ที่เป็นค่าบวกลวง```json
{
"description": "Keys that should remain untranslated...",
"keys": [
"common.model",
"common.oauth",
"health.cpu",
...
]
}

````

**มีอะไรอยู่ที่นี่:**

- ชื่อแบรนด์/ผลิตภัณฑ์: `landing.brandName`, `common.social-github`
- คำศัพท์ทางเทคนิค/ตัวย่อ: `health.cpu`, `mcpDashboard.pid`, `settings.ai`
- ICU/สตริงรูปแบบ: `apiManager.modelsCount`, `health.millisecondsShort`
- ค่าตัวยึดตำแหน่ง: `providers.openaiBaseUrlPlaceholder`, `cliTools.baseUrlPlaceholder`
- ชื่อโปรโตคอล: `common.http`, `common.oauth`, `providers.oauth2Label`
- ส่วนการนำทาง: `sidebar.primarySection`, `sidebar.cliSection`

**วิธีเพิ่มคีย์:**แก้ไขอาร์เรย์ `keys` ใน `scripts/i18n/untranslatable-keys.json` และดำเนินการตรวจสอบอีกครั้ง## CI Integration

### GitHub Actions (`.github/workflows/ci.yml`)

ไปป์ไลน์ CI ตรวจสอบสถานที่ทั้งหมดในทุก ๆ การพุชและ PR:

1.**`งาน i18n-matrix`**— ค้นหาไฟล์โลแคลทั้งหมดแบบไดนามิก (ยกเว้น `en.json`)
2.**`i18n` job**— รัน `validate_translation.py quick -l '<lang>''` สำหรับแต่ละภาษาในแบบคู่ขนาน
3.**`งาน ci-summary`**— รวมผลลัพธ์ไว้ในสรุปแดชบอร์ด```yaml
# i18n-matrix: discovers languages
LANGS=$(ls src/i18n/messages/*.json | xargs -n1 basename | sed 's/.json$//' | grep -v '^en$')

# i18n: validates each language
python3 scripts/validate_translation.py quick -l '${{ matrix.lang }}'
````

## Best Practices

### When Editing Translations

1.**แก้ไข `en.json` ก่อนเสมอ**— นั่นคือที่มาของความจริง
2.**เรียกใช้ `generate-multilang.mjs Messages`**เพื่อเผยแพร่คีย์ใหม่ไปยังสถานที่ทั้งหมด
3.**ตรวจสอบการแปลอัตโนมัติ**— Google Translate เป็นจุดเริ่มต้น ไม่ใช่จุดสิ้นสุด
4.**ตรวจสอบก่อนยืนยัน**— `python3 scripts/validate_translation.py quick -l <lang>`
5.**อัปเดต `untranslatable-keys.json`**หากคีย์ยังคงเป็นภาษาอังกฤษ### Placeholder Safety

- ตัวยึดตำแหน่ง ICU (`{count}`, `{value}`, `{total}`, `{seconds}`) จะต้องได้รับการเก็บรักษาไว้อย่างถูกต้อง
- รูปแบบพหูพจน์ (`{count, พหูพจน์, หนึ่ง {# model} other {# models}}`) จะต้องคงโครงสร้างไว้
- เครื่องมือตรวจสอบความถูกต้องจะตรวจจับตัวยึดตำแหน่งที่ไม่ตรงกันโดยอัตโนมัติ### Adding New Translation Keys in Code

```tsx
// Use namespaced keys
const t = useTranslations("settings");
t("cacheSettings"); // maps to settings.cacheSettings in JSON

// Run check_translations.py to verify keys exist
python3 scripts/check_translations.py --verbose
````

### RTL Considerations

- ภาษาอาหรับ (`ar`) และฮีบรู (`he`) เป็นภาษา RTL
- หลีกเลี่ยง CSS `left`/`right` แบบฮาร์ดโค้ด — ใช้คุณสมบัติเชิงตรรกะ `start`/`end`
- Visual QA จับเค้าโครง RTL ที่ไม่ตรงกันผ่าน `run-visual-qa.mjs`## Known Issues & History

### `in.json` → `hi.json` Fix

เดิมทีเครื่องกำเนิดใช้ `code: "in"` (โค้ด Google Translate ที่เลิกใช้แล้ว) สำหรับภาษาฮินดี แทนที่จะเป็น ISO 639-1 `hi` ที่ถูกต้อง สิ่งนี้ทำให้เกิด `in.json` ที่ซ้ำกันของ `hi.json` ที่ถูกละเลย แก้ไขโดยการเปลี่ยน `code: "in"` เป็น `code: "hi"` ใน `generate-multilang.mjs` และลบไฟล์ที่ถูกโยงถึงออก### `docs/i18n/README.md` Is Auto-Generated
