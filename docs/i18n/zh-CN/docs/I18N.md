### Source of Truth

-**UI 字符串**： `src/i18n/messages/en.json` （英文源，约 2800 个键）-**区域设置文件**：`src/i18n/messages/{locale}.json`（30 个翻译）-**框架**：具有基于 cookie 的区域设置解析的“next-intl” -**Config**: `src/i18n/config.ts` — 定义所有 30 个语言环境、语言名称、标志### Runtime Flow

1. 用户选择语言 → `NEXT_LOCALE` cookie 设置
2. `src/i18n/request.ts` 解析语言环境：cookie → `Accept-Language` 标头 → 后备 `en` 3.动态导入加载 `messages/{locale}.json`
3. 组件使用 `useTranslations("namespace")` 和 `t("key")`### Supported Locales

### 1. Register the Locale

编辑`src/i18n/config.ts`：```ts
// Add to LOCALES array
"xx",
// Add to LANGUAGES array
{ code: "xx", label: "XX", name: "Language Name", flag: "🏳️" },

````

### 2. Add to Generator

编辑 `scripts/i18n/generate-multilang.mjs` — 添加条目到 `LOCALE_SPECS`：```js
{
  code: "xx",
  googleTl: "xx",
  label: "XX",
  flag: "🏳️",
  languageName: "Language Name",
  readmeName: "Language Name",
  docsName: "Language Name",
},
````

### 3. Generate Initial Translation

```bash
node scripts/i18n/generate-multilang.mjs messages
```

这将创建通过谷歌翻译从“en.json”自动翻译的“src/i18n/messages/xx.json”。### 4. Review & Fix Auto-Translations

自动翻译是一个起点。手动审核：

- 技术准确性
- 适合上下文的术语
- 正确处理占位符（“{count}”、“{value}”等）### 5. Validate

```bash
python3 scripts/validate_translation.py quick -l xx
python3 scripts/validate_translation.py diff common -l xx
```

### 6. Generate Translated Documentation

```bash
node scripts/i18n/generate-multilang.mjs docs
```

## Auto-Translation Pipeline

### generate-multilang.mjs (Google Translate)

**主要自动翻译引擎**— 使用 Google Translate 免费 API 生成 UI 字符串、自述文件和文档的翻译。```bash
node scripts/i18n/generate-multilang.mjs [messages|readme|docs|all]

``````

|模式|它有什么作用 |
| ---------- | ------------------------------------------------------------------------------------------ |
| `消息` |从 `en.json` 翻译 `src/i18n/messages/{locale}.json` 中缺少的键 |
| `自述文件` |将 `README.md` 翻译为项目根目录下的所有语言环境中的 `README.{code}.md` |
| `文档` |将 `DOC_SOURCE_FILES` 翻译为 `docs/i18n/{locale}/{docName}` |
| `全部` |运行所有三种模式 |

**特点：**

-**文本保护**：在翻译前屏蔽代码块（`````）、内联代码（`````）、markdown 链接/图像（`[text](url)`）、HTML 标签、表格和 ICU 占位符（`{count}`、`{value}`、`{total}` 等），然后恢复它们
-**分块批处理**：使用“__OMNIROUTE_I18N_SEPARATOR__”分隔符连接多个字符串，以最大程度地减少 API 调用（每个请求最多 1800 个字符）
-**内存缓存**：避免会话中重复字符串的冗余 API 调用
-**重试逻辑**：针对 429/5xx 错误的指数退避（最多 5 次尝试，300 毫秒 × 尝试延迟）
-**超时**：每个请求 20 秒
-**跳过现有**：如果目标文件已存在，则不会覆盖它

**重要行为：**

- `docs/i18n/README.md` 每次运行都会**重新生成**— 它是所有文档的自动生成索引
- 根“README.{code}.md”文件仅在不存在时创建（跳过“EXISTING_README_CODES”中的语言环境）
- 语言栏（`🌐**语言：**...`）会自动插入/更新所有翻译文档中### i18n_autotranslate.py (LLM-based)

**辅助翻译器**— 使用任何 OpenAI 兼容的 LLM API（包括 OmniRoute 本身）来翻译现有的 `docs/i18n/` markdown 文件。最适合以比 Google 翻译更好的质量润色或重新翻译文档。```bash
python3 scripts/i18n_autotranslate.py \
  --api-url http://localhost:20128/v1 \
  --api-key sk-your-key \
  --model gpt-4o
``````

**特点：**

- 扫描“docs/i18n/” markdown 文件中的英文段落
- 跳过代码块、表格和已翻译的内容
- 通过技术翻译系统提示将段落发送给法学硕士
- 支持全部 30 种语言## Validation & QA

### validate_translation.py

**翻译验证器**— 将任何语言环境 JSON 与“en.json”进行比较并报告问题。```bash

# Quick check (counts only)

python3 scripts/validate_translation.py quick -l cs

# Output:

# Missing: 0

# Untranslated: 0

# Ignored (UNTRANSLATABLE_KEYS): 236

# Detailed diff by category

python3 scripts/validate_translation.py diff common -l cs
python3 scripts/validate_translation.py diff settings -l cs

# Export to CSV

python3 scripts/validate_translation.py csv -l cs > report.csv

# Export to Markdown

python3 scripts/validate_translation.py md -l cs > report.md

# Full report (default)

python3 scripts/validate_translation.py -l cs

````

**检测：**

-**缺少键**— `en.json` 中的键，但不在语言环境文件中
-**额外键**— 语言环境文件中的键，但不在 `en.json` 中
-**未翻译的键**— 区域设置值等于英文源的键（不包括白名单）
-**占位符不匹配**— ICU 占位符在源代码和翻译之间不匹配

**退出代码：**
|代码|意义|
|------|---------|
| 0 |好的 |
| 1 |一般错误 |
| 2 |缺少字符串（硬错误）|
| 3 |未翻译的警告（软）|

**环境：**设置 `TRANSLATION_LANG=cs` 或使用 `-l cs` 标志。### check_translations.py

**代码到 JSON 密钥检查器**— 扫描 `src/**/*.tsx` 和 `src/**/*.ts` 中的 `useTranslations()` 调用，并验证 `en.json` 中是否存在所有引用的密钥。```bash
# Basic check
python3 scripts/check_translations.py

# Verbose output
python3 scripts/check_translations.py --verbose

# Auto-fix (adds missing keys to en.json)
python3 scripts/check_translations.py --fix
````

### generate-qa-checklist.mjs

**静态分析 QA**— 扫描 Next.js 页面文件中的 i18n 风险指标并生成 Markdown 报告。```bash
node scripts/i18n/generate-qa-checklist.mjs

````

**检查：**

- 固定宽度类的使用（溢出风险）
- 定向左/右类别（RTL 风险）
- 容易被剪裁的图案
- 区域设置奇偶校验（缺少/额外的键与“en.json”）
- 优先区域设置中的自述文件语言选择器栏（`es`、`fr`、`de`、`ja`、`ar`）

**输出：**`docs/reports/i18n-qa-checklist-{date}.md`### run-visual-qa.mjs

**通过 Playwright 进行视觉 QA**— 在多个区域设置和视口中截取所有仪表板路由的屏幕截图，然后评估页面运行状况。```bash
# Default: es, fr, de, ja, ar on localhost:20128
node scripts/i18n/run-visual-qa.mjs

# Custom base URL and locales
QA_BASE_URL=http://staging.example.com QA_LOCALES=de,fr node scripts/i18n/run-visual-qa.mjs

# Custom routes
QA_ROUTES=/dashboard/settings,/dashboard/providers node scripts/i18n/run-visual-qa.mjs
````

**检测：**

- 文本溢出
- 元素剪辑
- RTL布局不匹配

**输出：**`docs/reports/i18n-visual-qa-{date}.md` + JSON 报告## Managing Untranslatable Keys

### untranslatable-keys.json

**文件：**`scripts/i18n/untranslatable-keys.json`

应与英文源保持相同的密钥白名单。由 `validate_translation.py` 使用以避免误报“未翻译”警告。```json
{
"description": "Keys that should remain untranslated...",
"keys": [
"common.model",
"common.oauth",
"health.cpu",
...
]
}

````

**这里属于什么：**

- 品牌/产品名称：`landing.brandName`、`common.social-github`
- 技术术语/缩写词：`health.cpu`、`mcpDashboard.pid`、`settings.ai`
- ICU/格式字符串：`apiManager.modelsCount`、`health.millisecondsShort`
- 占位符值：`providers.openaiBaseUrlPlaceholder`、`cliTools.baseUrlPlaceholder`
- 协议名称：`common.http`、`common.oauth`、`providers.oauth2Label`
- 导航部分：`sidebar.primarySection`、`sidebar.cliSection`

**要添加密钥：**编辑 `scripts/i18n/untranslatable-keys.json` 中的 `keys` 数组并重新运行验证。## CI Integration

## Best Practices

### When Editing Translations

1.**始终首先编辑`en.json`**- 这是事实的来源
2.**运行 `generate-multilang.mjs messages`**将新密钥传播到所有语言环境
3.**查看自动翻译**- 谷歌翻译是一个起点，而不是最终结果
4.**提交前验证**— `python3 script/validate_translation.py fast -l <lang>`
5.**如果密钥应保留英文，则更新 `untranslatable-keys.json`**### Placeholder Safety

- ICU 占位符（“{count}”、“{value}”、“{total}”、“{seconds}”）必须准确保留
- 复数格式（`{count,plural, one {# model} other {# models}}`）必须保持结构
- 验证器自动检测占位符不匹配### Adding New Translation Keys in Code

```tsx
// Use namespaced keys
const t = useTranslations("settings");
t("cacheSettings"); // maps to settings.cacheSettings in JSON

// Run check_translations.py to verify keys exist
python3 scripts/check_translations.py --verbose
````

### RTL Considerations

- 阿拉伯语 (`ar`) 和希伯来语 (`he`) 是 RTL 语言环境
- 避免硬编码 `left`/`right` CSS — 使用 `start`/`end` 逻辑属性
- Visual QA 通过“run-visual-qa.mjs”捕获 RTL 布局不匹配## Known Issues & History

### `in.json` → `hi.json` Fix

生成器最初使用印地语的 `code: "in"`（已弃用的 Google 翻译代码），而不是正确的 ISO 639-1 `hi`。这创建了“hi.json”的孤立“in.json”副本。通过将 `generate-multilang.mjs` 中的 `code: "in"` 更改为 `code: "hi"` 并删除孤立文件来修复。### `docs/i18n/README.md` Is Auto-Generated

`docs/i18n/README.md` 文件由 `generate-multilang.mjs docs` 完全重新生成。任何手动编辑都将丢失。使用“docs/I18N.md”（此文件）来保存应保留的手写文档。### External Untranslatable Keys List

“untranslatable-keys.json”白名单已从“validate_translation.py”中的内联 Python 集移至外部 JSON 文件，以便于维护。验证器在运行时加载它。### `generate-multilang.mjs` Hindi Code Fix

生成器最初使用印地语的 `code: "in"`（已弃用的 Google 翻译代码），而不是正确的 ISO 639-1 `hi`。这是由“diegosouzapw”在上游提交“952b0b22c”中引入的。通过将 `LOCALE_SPECS` 数组中的 `code: "in"` 更改为 `code: "hi"` 并删除孤立的 `in.json` 文件来修复。### `validate_translation.py` Ignored Count Output

“快速”检查现在显示“untranslatable-keys.json”中忽略的键的计数：```
