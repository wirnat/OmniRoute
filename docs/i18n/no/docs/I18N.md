### Source of Truth

-**UI-strenger**: `src/i18n/messages/en.json` (engelsk kilde, ~2800 nøkler) -**Lokale filer**: `src/i18n/messages/{locale}.json` (30 oversettelser) -**Rammeverk**: `neste-intl` med informasjonskapselbasert lokaloppløsning -**Config**: `src/i18n/config.ts` — definerer alle 30 lokaliteter, språknavn, flagg### Runtime Flow

1. Brukeren velger språk → `NEXT_LOCALE` informasjonskapselsett
2. `src/i18n/request.ts` løser lokalitet: informasjonskapsel → `Accept-Language` header → fallback `no`
3. Dynamisk import laster inn `messages/{locale}.json`
4. Komponenter bruker `useTranslations("namespace")` og `t("key")`### Supported Locales

### 1. Register the Locale

Rediger `src/i18n/config.ts`:```ts
// Add to LOCALES array
"xx",
// Add to LANGUAGES array
{ code: "xx", label: "XX", name: "Language Name", flag: "🏳️" },

````

### 2. Add to Generator

Rediger `scripts/i18n/generate-multilang.mjs` — legg til oppføring i `LOCALE_SPECS`:```js
{
  code: "xx",
  googleTl: "xx",
  label: "XX",
  flag: "🏳️",
  languageName: "Language Name",
  readmeName: "Language Name",
  docsName: "Language Name",
},
````

### 3. Generate Initial Translation

```bash
node scripts/i18n/generate-multilang.mjs messages
```

Dette oppretter `src/i18n/messages/xx.json` automatisk oversatt fra `en.json` via Google Translate.### 4. Review & Fix Auto-Translations

Automatiske oversettelser er et utgangspunkt. Gjennomgå manuelt for:

- Teknisk nøyaktighet
- Konteksttilpasset terminologi
  – Riktig håndtering av plassholdere (`{count}`, `{value}` osv.)### 5. Validate

```bash
python3 scripts/validate_translation.py quick -l xx
python3 scripts/validate_translation.py diff common -l xx
```

### 6. Generate Translated Documentation

```bash
node scripts/i18n/generate-multilang.mjs docs
```

## Auto-Translation Pipeline

### generate-multilang.mjs (Google Translate)

**Primær motor for automatisk oversettelse**— bruker Google Translate gratis API for å generere oversettelser for UI-strenger, README-er og dokumentasjon.```bash
node scripts/i18n/generate-multilang.mjs [messages|readme|docs|all]

````

| Modus | Hva den gjør |
| ---------- | ------------------------------------------------------------------------------------ |
| `meldinger` | Oversetter manglende nøkler i `src/i18n/messages/{locale}.json` fra `en.json` |
| `les meg` | Oversetter `README.md` til alle lokaliteter som `README.{code}.md` i prosjektroten |
| `dokumenter` | Oversetter `DOC_SOURCE_FILES` til `docs/i18n/{locale}/{docName}` |
| `alle` | Kjører alle tre modusene |

**Funksjoner:**

-**Tekstbeskyttelse**: Maskerer kodeblokker (` ``` `), innebygd kode (`` ```), markdown-lenker/bilder (`[text](url)`), HTML-koder, tabeller og ICU-plassholdere (`{count}`, `{value}`, `{total}` osv.) før oversettelse, og gjenoppretter dem deretter
-**Chunked batching**: Slår sammen flere strenger med `__OMNIROUTE_I18N_SEPARATOR__` skilletegn for å minimere API-kall (maks. 1800 tegn per forespørsel)
-**Cache i minnet**: Unngår redundante API-kall for gjentatte strenger i en økt
-**Prøv logikk på nytt**: Eksponentiell backoff (opptil 5 forsøk med 300ms × forsøksforsinkelse) for 429/5xx feil
-**Tidsavbrudd**: 20 sekunder per forespørsel
-**Hopp over eksisterende**: Hvis målfilen allerede eksisterer, blir den IKKE overskrevet

**Viktig oppførsel:**

- `docs/i18n/README.md`**regenereres**hver kjøring – det er en automatisk generert indeks over alle dokumenter
- Root `README.{code}.md`-filer opprettes bare hvis de ikke eksisterer (hopper over lokaliteter i `EXISTING_README_CODES`)
- Språklinjer (`🌐**Språk:**...`) settes inn/oppdateres automatisk i alle oversatte dokumenter### i18n_autotranslate.py (LLM-based)

**Sekundær oversetter**— bruker en hvilken som helst OpenAI-kompatibel LLM API (inkludert OmniRoute selv) for å oversette eksisterende `docs/i18n/` markdown-filer. Best for polering eller re-oversettelse av dokumenter med bedre kvalitet enn Google Translate.```bash
python3 scripts/i18n_autotranslate.py \
  --api-url http://localhost:20128/v1 \
  --api-key sk-your-key \
  --model gpt-4o
````

**Funksjoner:**

- Skanner `docs/i18n/` markdown-filer for engelske avsnitt
- Hopper over kodeblokker, tabeller og allerede oversatt innhold
- Sender avsnitt til LLM med teknisk oversettelsessystemspørsmål
- Støtter alle 30 språk## Validation & QA

### validate_translation.py

**Oversettelsesvalidator**— sammenligner enhver lokal JSON med 'en.json' og rapporterer problemer.```bash

# Quick check (counts only)

python3 scripts/validate_translation.py quick -l cs

# Output:

# Missing: 0

# Untranslated: 0

# Ignored (UNTRANSLATABLE_KEYS): 236

# Detailed diff by category

python3 scripts/validate_translation.py diff common -l cs
python3 scripts/validate_translation.py diff settings -l cs

# Export to CSV

python3 scripts/validate_translation.py csv -l cs > report.csv

# Export to Markdown

python3 scripts/validate_translation.py md -l cs > report.md

# Full report (default)

python3 scripts/validate_translation.py -l cs

````

**Oppdager:**

-**Mangler nøkler**- nøkler i 'en.json' men ikke i lokalitetsfil
-**Ekstranøkler**- nøkler i lokalitetsfilen, men ikke i 'en.json'
-**Uoversatte nøkler**- nøkler der lokalverdi er lik engelsk kilde (unntatt tillatelsesliste)
-**Plassholdermismatches**— ICU-plassholdere som ikke samsvarer mellom kilde og oversettelse

**Utgangskoder:**
| Kode | Betydning |
|------|--------|
| 0 | OK |
| 1 | Generisk feil |
| 2 | Manglende strenger (hard feil) |
| 3 | Uoversatt advarsel (myk) |

**Miljø:**Angi `TRANSLATION_LANG=cs` eller bruk `-l cs`-flagget.### check_translations.py

**Code-to-JSON key checker**– skanner `src/**/*.tsx` og `src/**/*.ts` for `useTranslations()`-kall og bekrefter at alle refererte nøkler finnes i `en.json`.```bash
# Basic check
python3 scripts/check_translations.py

# Verbose output
python3 scripts/check_translations.py --verbose

# Auto-fix (adds missing keys to en.json)
python3 scripts/check_translations.py --fix
````

**Statisk analyse QA**— skanner Next.js-sidefiler for i18n-risikoberegninger og genererer en Markdown-rapport.```bash

### untranslatable-keys.json

**Fil:**`scripts/i18n/untranslatable-keys.json`

Tillatelsesliste over nøkler som skal forbli identiske med engelsk kilde. Brukes av `validate_translation.py` for å unngå falske positive "uoversatte" advarsler.```json
{
"description": "Keys that should remain untranslated...",
"keys": [
"common.model",
"common.oauth",
"health.cpu",
...
]
}

````

**Hva hører hjemme her:**

– Merke-/produktnavn: `landing.brandName`, `common.social-github`
- Tekniske termer/akronymer: "health.cpu", "mcpDashboard.pid", "settings.ai".
- ICU/formatstrenger: `apiManager.modelsCount`, `health.millisecondsShort`
– Plassholderverdier: `providers.openaiBaseUrlPlaceholder`, `cliTools.baseUrlPlaceholder`
– Protokollnavn: `common.http`, `common.oauth`, `providers.oauth2Label`
- Navigasjonsseksjoner: `sidebar.primarySection`, `sidebar.cliSection`

**For å legge til en nøkkel:**Rediger `keys`-arrayen i `scripts/i18n/untranslatable-keys.json` og kjør valideringen på nytt.## CI Integration

## Best Practices

### When Editing Translations

1.**Rediger alltid `en.json` først**– det er kilden til sannheten
2.**Kjør `generate-multilang.mjs messages`**for å spre nye nøkler til alle lokaliteter
3.**Gjennomgå automatiske oversettelser**— Google Translate er et utgangspunkt, ikke endelig
4.**Valider før du forplikter**— `python3 scripts/validate_translation.py quick -l <lang>`
5.**Oppdater `untranslatable-keys.json`**hvis en nøkkel skal forbli på engelsk### Placeholder Safety

- ICU plassholdere (`{count}`, `{value}`, `{total}`, `{seconds}`) må bevares nøyaktig
– Flertallsformater («{count, plural, one {# model} other {# models}}») må opprettholde struktur
- Validatoren oppdager plassholderfeil automatisk### Adding New Translation Keys in Code

```tsx
// Use namespaced keys
const t = useTranslations("settings");
t("cacheSettings"); // maps to settings.cacheSettings in JSON

// Run check_translations.py to verify keys exist
python3 scripts/check_translations.py --verbose
````

### RTL Considerations

- Arabisk (`ar`) og hebraisk (`he`) er RTL-lokaliteter
- Unngå hardkodet "venstre"/"høyre" CSS - bruk "start"/"slutt" logiske egenskaper
- Visual QA fanger opp uoverensstemmelser i RTL-layout via `run-visual-qa.mjs`## Known Issues & History

### `in.json` → `hi.json` Fix

Generatoren brukte opprinnelig `code: "in"` (avviklet Google Translate-kode) for hindi i stedet for den korrekte ISO 639-1 `hi`. Dette opprettet en foreldreløs "in.json"-duplikat av "hi.json". Rettet ved å endre `code: "in"` til `code: "hi"` i `generate-multilang.mjs` og fjerne den foreldreløse filen.### `docs/i18n/README.md` Is Auto-Generated

Filen `docs/i18n/README.md` er fullstendig regenerert av `generate-multilang.mjs docs`. Eventuelle manuelle redigeringer vil gå tapt. Bruk `docs/I18N.md` (denne filen) for håndskrevet dokumentasjon som skal vedvare.### External Untranslatable Keys List

Tillatelseslisten `untranslatable-keys.json` ble flyttet fra et innebygd Python-sett i `validate_translation.py` til en ekstern JSON-fil for enklere vedlikehold. Validatoren laster den under kjøretid.### `generate-multilang.mjs` Hindi Code Fix

Generatoren brukte opprinnelig `code: "in"` (avviklet Google Translate-kode) for hindi i stedet for den korrekte ISO 639-1 `hi`. Dette ble introdusert i oppstrøms commit `952b0b22c` av `diegosouzapw`. Rettet ved å endre `code: "in"` til `code: "hi"` i `LOCALE_SPECS`-matrisen og fjerne den foreldreløse `in.json`-filen.### `validate_translation.py` Ignored Count Output
