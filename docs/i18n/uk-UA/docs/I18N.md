### Source of Truth

-**Рядки інтерфейсу користувача**: `src/i18n/messages/en.json` (англомовне джерело, ~2800 ключів) -**Локальні файли**: `src/i18n/messages/{locale}.json` (30 перекладів) -**Framework**: `next-intl` із розпізнаванням локалі на основі файлів cookie -**Конфігурація**: `src/i18n/config.ts` — визначає всі 30 локалей, назви мов, прапори### Runtime Flow

1. Користувач вибирає мову → набір файлів cookie `NEXT_LOCALE`
2. `src/i18n/request.ts` визначає локаль: cookie → заголовок `Accept-Language` → запасний `en`
3. Динамічний імпорт завантажує `messages/{locale}.json`
4. Компоненти використовують `useTranslations("namespace")` і `t("key")`### Supported Locales

### 1. Register the Locale

Відредагуйте `src/i18n/config.ts`:```ts
// Add to LOCALES array
"xx",
// Add to LANGUAGES array
{ code: "xx", label: "XX", name: "Language Name", flag: "🏳️" },

````

### 2. Add to Generator

Відредагуйте `scripts/i18n/generate-multilang.mjs` — додайте запис до `LOCALE_SPECS`:```js
{
  code: "xx",
  googleTl: "xx",
  label: "XX",
  flag: "🏳️",
  languageName: "Language Name",
  readmeName: "Language Name",
  docsName: "Language Name",
},
````

### 3. Generate Initial Translation

```bash
node scripts/i18n/generate-multilang.mjs messages
```

Це створює автоматичний переклад `src/i18n/messages/xx.json` з `en.json` за допомогою Перекладача Google.### 4. Review & Fix Auto-Translations

Автоматичні переклади є відправною точкою. Перегляньте вручну для:

- Технічна точність
- Термінологія, що відповідає контексту
- Правильна обробка заповнювачів (`{count}`, `{value}` тощо)### 5. Validate

```bash
python3 scripts/validate_translation.py quick -l xx
python3 scripts/validate_translation.py diff common -l xx
```

### 6. Generate Translated Documentation

```bash
node scripts/i18n/generate-multilang.mjs docs
```

## Auto-Translation Pipeline

### generate-multilang.mjs (Google Translate)

**Основний механізм автоматичного перекладу**— використовує безкоштовний API Google Translate для створення перекладів рядків інтерфейсу користувача, файлів README та документації.```bash
node scripts/i18n/generate-multilang.mjs [messages|readme|docs|all]

````

| Режим | Що він робить |
| ---------- | ---------------------------------------------------------------------------------- |
| `повідомлення` | Перекладає відсутні ключі в `src/i18n/messages/{locale}.json` з `en.json` |
| `readme` | Перекладає `README.md` на всі локалі як `README.{code}.md` у корені проекту |
| `документи` | Перекладає `DOC_SOURCE_FILES` на `docs/i18n/{locale}/{docName}` |
| `усі` | Запускає всі три режими |

**Особливості:**

-**Захист тексту**: маскує блоки коду (` ``` `), вбудований код (`` ` ``), посилання/зображення розмітки (`[text](url)`), HTML-теги, таблиці та заповнювачі ICU (`{count}`, `{value}`, `{total}` тощо) перед перекладом, а потім відновлює їх
-**Пакетування на фрагменти**: об’єднує кілька рядків за допомогою роздільників `__OMNIROUTE_I18N_SEPARATOR__` для мінімізації викликів API (макс. 1800 символів на запит)
-**Кеш-пам’ять**: уникає надлишкових викликів API для повторюваних рядків протягом сеансу.
-**Логіка повторної спроби**: експоненціальна відстрочка (до 5 спроб із затримкою 300 мс × спроба) для помилок 429/5xx
-**Тайм-аут**: 20 секунд на запит
-**Пропустити існуючий**: якщо цільовий файл уже існує, він НЕ перезаписується

**Важлива поведінка:**

- `docs/i18n/README.md`**регенерується**кожного запуску — це автоматично згенерований індекс усіх документів
- Кореневі файли `README.{code}.md` створюються, лише якщо вони не існують (пропускає локалі в `EXISTING_README_CODES`)
- Мовні панелі (`🌐**Мови:**...`) автоматично вставляються/оновлюються в усі перекладені документи### i18n_autotranslate.py (LLM-based)

**Вторинний перекладач**— використовує будь-який OpenAI-сумісний LLM API (включаючи сам OmniRoute) для перекладу існуючих файлів розмітки `docs/i18n/`. Найкраще підходить для вдосконалення чи повторного перекладу документів із кращою якістю, ніж Перекладач Google.```bash
python3 scripts/i18n_autotranslate.py \
  --api-url http://localhost:20128/v1 \
  --api-key sk-your-key \
  --model gpt-4o
````

**Особливості:**

— Сканує файли розмітки `docs/i18n/` на наявність абзаців англійською мовою

- Пропускає блоки коду, таблиці та вже перекладений вміст
- Надсилає параграфи до LLM із підказкою системи технічного перекладу
- Підтримує всі 30 мов## Validation & QA

### validate_translation.py

**Засіб перевірки перекладу**— порівнює будь-які локалі JSON із `en.json` і повідомляє про проблеми.```bash

# Quick check (counts only)

python3 scripts/validate_translation.py quick -l cs

# Output:

# Missing: 0

# Untranslated: 0

# Ignored (UNTRANSLATABLE_KEYS): 236

# Detailed diff by category

python3 scripts/validate_translation.py diff common -l cs
python3 scripts/validate_translation.py diff settings -l cs

# Export to CSV

python3 scripts/validate_translation.py csv -l cs > report.csv

# Export to Markdown

python3 scripts/validate_translation.py md -l cs > report.md

# Full report (default)

python3 scripts/validate_translation.py -l cs

````

**Виявляє:**

-**Відсутні ключі**— ключі в `en.json`, але не у файлі локалі
-**Додаткові ключі**— ключі у файлі локалі, але не в `en.json`
-**Неперекладені ключі**— ключі, де значення локалі дорівнює англійському джерелу (за винятком білого списку)
-**Placeholder mismatches**— заповнювачі ICU, які не збігаються між джерелом і перекладом

**Коди виходу:**
| Код | Значення |
|------|---------|
| 0 | ОК |
| 1 | Загальна помилка |
| 2 | Відсутні рядки (серйозна помилка) |
| 3 | Неперекладене попередження (м'яке) |

**Середовище:**Встановіть `TRANSLATION_LANG=cs` або використовуйте прапор `-l cs`.### check_translations.py

**Засіб перевірки ключа Code-to-JSON**— сканує `src/**/*.tsx` і `src/**/*.ts` на предмет викликів `useTranslations()` і перевіряє наявність усіх ключів, на які посилаються, у `en.json`.```bash
# Basic check
python3 scripts/check_translations.py

# Verbose output
python3 scripts/check_translations.py --verbose

# Auto-fix (adds missing keys to en.json)
python3 scripts/check_translations.py --fix
````

node scripts/i18n/generate-qa-checklist.mjs

````

**Чеки:**

- Використання класу фіксованої ширини (ризик переповнення)
- Спрямовані ліворуч/праворуч класи (ризик RTL)

**Вихід:**`docs/reports/i18n-qa-checklist-{date}.md`### run-visual-qa.mjs

**Візуальна перевірка якості через Playwright**— робить знімки екрана всіх маршрутів інформаційної панелі в різних регіонах і вікнах перегляду, а потім оцінює стан сторінки.```bash

**Виявляє:**

- Переповнення тексту
- Відсікання елементів
- Невідповідність макета RTL

**Вихід:**`docs/reports/i18n-visual-qa-{date}.md` + звіт JSON## Managing Untranslatable Keys

### untranslatable-keys.json

**Файл:**`scripts/i18n/untranslatable-keys.json`

Білий список ключів, які повинні залишатися ідентичними англійському джерелу. Використовується `validate_translation.py`, щоб уникнути хибно-позитивних попереджень про "неперекладені".```json
{
"description": "Keys that should remain untranslated...",
"keys": [
"common.model",
"common.oauth",
"health.cpu",
...
]
}

````

**Що тут належить:**

- Назви брендів/продуктів: `landing.brandName`, `common.social-github`
- Технічні терміни/акроніми: `health.cpu`, `mcpDashboard.pid`, `settings.ai`
- ICU/рядки формату: `apiManager.modelsCount`, `health.millisecondsShort`
- Значення заповнювача: `providers.openaiBaseUrlPlaceholder`, `cliTools.baseUrlPlaceholder`
- Назви протоколів: `common.http`, `common.oauth`, `providers.oauth2Label`
- Розділи навігації: `sidebar.primarySection`, `sidebar.cliSection`

**Щоб додати ключ:**Відредагуйте масив `keys` у `scripts/i18n/untranslatable-keys.json` і повторно запустіть перевірку.## CI Integration

## Best Practices

### When Editing Translations

1.**Завжди спочатку редагуйте `en.json`**— це джерело правди
2.**Запустіть `generate-multilang.mjs messages`**, щоб поширити нові ключі до всіх локалей
3.**Переглянути автоматичні переклади**— Перекладач Google є початковою, а не остаточною точкою
4.**Перевірити перед фіксацією**— `python3 scripts/validate_translation.py quick -l <lang>`
5.**Оновіть `untranslatable-keys.json`**, якщо ключ має залишитися англійською### Placeholder Safety

- Заповнювачі ICU (`{count}`, `{value}`, `{total}`, `{seconds}`) мають бути точно збережені
- Формати множини (`{count, plural, one {# model} other {# models}}`) мають підтримувати структуру
- Валідатор автоматично виявляє невідповідності заповнювачів### Adding New Translation Keys in Code

```tsx
// Use namespaced keys
const t = useTranslations("settings");
t("cacheSettings"); // maps to settings.cacheSettings in JSON

// Run check_translations.py to verify keys exist
python3 scripts/check_translations.py --verbose
````

### RTL Considerations

- Арабська (`ar`) та іврит (`he`) є мовами RTL
- Уникайте жорстко закодованого CSS `left`/`right` — використовуйте логічні властивості `start`/`end`
- Visual QA виявляє невідповідності макета RTL за допомогою `run-visual-qa.mjs`## Known Issues & History

### `in.json` → `hi.json` Fix

Генератор спочатку використовував `code: "in"` (застарілий код Google Translate) для гінді замість правильного ISO 639-1 `hi`. Це створило загублену копію `in.json` `hi.json`. Виправлено шляхом зміни `code: "in"` на `code: "hi"` у `generate-multilang.mjs` і видалення загубленого файлу.### `docs/i18n/README.md` Is Auto-Generated

Файл `docs/i18n/README.md` повністю генерується за допомогою `generate-multilang.mjs docs`. Усі правки, внесені вручну, буде втрачено. Використовуйте `docs/I18N.md` (цей файл) для рукописної документації, яка має зберігатися.### External Untranslatable Keys List

Для спрощення обслуговування білий список `untranslatable-keys.json` було переміщено з вбудованого набору Python у `validate_translation.py` до зовнішнього файлу JSON. Валідатор завантажує його під час виконання.### `generate-multilang.mjs` Hindi Code Fix
